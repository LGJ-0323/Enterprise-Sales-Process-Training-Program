# 销售陪练系统 — 数据结构改进方案

> 基于 `docs/` 下三个 JSONL 文件的逐项审计，以及模型运行时集成方案。
>
> 编写日期：2026-06-30

---

## 一、现有三个文件总览

| 文件 | 层级 | 当前量 | 核心用途 |
|---|---|---|---|
| `raw_calls.jsonl` | L1 通话转录层 | 1 条 | 原始对话文本 + 元数据，是一切数据的底层原料 |
| `roleplay_cases.jsonl` | L3 角色扮演配置层 | 1 条 | 驱动 AI 客户行为——告诉模型「这个客户是谁、怎么说话、什么情况下说什么」 |
| `evaluation_rubrics.jsonl` | L4 评分标准层 | 1 条 | 训练结束后的评分维度和标准 |

---

## 二、`raw_calls.jsonl` 审计与改进

### 2.1 缺失项

| 问题 | 影响 | 严重度 |
|---|---|---|
| 没有 `intent`（话语意图标签） | 无法检索「处理价格异议的对话」，无法做模式分析 | 🔴 高 |
| 没有 `tone`（语气标签） | 「没事」= 冷漠还是配合？模型只看文字学不到语调 | 🔴 高 |
| 没有 `result_label`（结构化结果标签） | 无法查询「哪些电话拿下了下一步」 | 🔴 高 |
| 没有 `preceding_context`（前置上下文） | 只知道昨天报过价，昨天具体说了什么全部丢失 | 🟡 中 |
| 没有 `state_trace`（每轮状态） | 无法追踪状态变化轨迹，不能用于状态机校准 | 🟡 中 |
| 没有 `sales_metadata`（销售属性） | 无法区分新人通话和老手通话 | 🟢 低 |

### 2.3 intent 标签建议值

| intent | 含义 | 示例 |
|---|---|---|
| `greeting` | 打招呼 | "喂，你好" |
| `acknowledge_call` | 接起电话的简短确认 | "没事，你说" |
| `confirm_status` | 确认当前状态 | "那现在就是等客户回复？" |
| `explain_policy` | 解释规则/费用 | "超重费是UPS末端规则" |
| `probe_need` | 挖掘需求/标准 | "您选货代有什么要求？" |
| `handle_objection` | 处理异议 | "其他家可能把费用放单价里" |
| `push_next_step` | 推进下一步 | "可以拜访您一下吗？" |
| `capture_info` | 获取客户信息 | "您贵姓？" |
| `present_value` | 介绍公司优势 | "我们是一手货代，只接普货" |
| `deflect` | 回避/推责（负面） | "这个没办法，规则就是这样" |
| `hard_close` | 强逼单（负面） | "那你今天能定吗？" |
| `build_rapport` | 建立关系 | "您也在合肥？那挺方便的" |

### 2.4 tone 标签建议值

| tone | 含义 |
|---|---|
| `neutral` | 中性 |
| `slightly_cold` | 微冷淡 |
| `warming_up` | 正在回暖 |
| `cooperative` | 配合 |
| `impatient` | 不耐烦 |
| `skeptical` | 怀疑 |
| `interested` | 表现出兴趣 |
| `dismissive` | 敷衍 |

## 三、`roleplay_cases.jsonl` 审计与改进

### 3.1 缺失项

| 问题 | 影响 | 严重度 |
|---|---|---|
| 行为规则是静态列表，没有状态驱动 | 客户在不同状态下的行为无法区分 | 🔴 高 |
| 没有 `state_machine`（状态流转图） | 客户什么时候从 guarded 进入 warming_up？无法控制 | 🔴 高 |
| 没有 `few_shot_examples` | 规则再多，不如 3 个真实对话片段让模型理解「这个客户怎么说话」 | 🔴 高 |
| 没有 `failure_conditions` | 不知道什么情况算训练失败 | 🟡 中 |
| 没有 `difficulty_variants` | 同一个 case 在 easy/normal/hard 下行为应该不同 | 🟡 中 |
| training_goals 和 rubric 维度无关联 | 训练目标和评分标准脱节 | 🟢 低 |

## 四、`evaluation_rubrics.jsonl` 审计与改进

### 4.1 缺失项

| 问题 | 影响 | 严重度 |
|---|---|---|
| 评分标准是定性的，没有 `observable_evidence` | 「能说明超重费不是随意加价」——判分时没有依据，容易主观 | 🔴 高 |
| 维度权重是固定的 | 开场在陌 call 里权重 25，在逼单里权重 5。但现在所有场景共用一份权重 | 🔴 高 |
| 没有 `applicability_condition` | 这通电话如果根本没提超重费，这个维度怎么判？ | 🟡 中 |
| `ideal_sales_flow` 是线性列表 | 真实对话不是 1→2→3→4，是树状的 | 🟡 中 |
| 没有跨维度依赖 | 「需求挖掘」差 → 「价值匹配」必然差，但规则没体现 | 🟢 低 |
| 没有打分示例 | 只有文字标准，缺少具体的打分参考范例 | 🟢 低 |

### 5.3 Prompt 组装核心逻辑（伪代码）

```python
# core/prompt_assembler.py

import json
from pathlib import Path

def load_case(training_type: str, difficulty: str):
    """从 roleplay_cases.jsonl 匹配案例"""
    with open("docs/roleplay_cases.jsonl", encoding="utf-8") as f:
        for line in f:
            case = json.loads(line)
            if case["training_type"] == training_type and case["difficulty"] == difficulty:
                return case
    return None

def build_few_shot_string(case: dict, max_examples: int = 4):
    """将 few_shot 示例转为 prompt 文本"""
    examples = case.get("few_shot_examples", [])
    lines = ["以下是该类型客户的真实对话示例（请模仿这种语气和风格）："]
    for i, ex in enumerate(examples[:max_examples], 1):
        lines.append(
            f"示例{i}（{ex['state']}状态）：\n"
            f"  销售说：「{ex['sales_input']}」\n"
            f"  客户回：「{ex['customer_reply']}」\n"
            f"  要点：{ex['why']}"
        )
    return "\n".join(lines)

def assemble_prompt(case: dict, current_state: str, history: list) -> str:
    """组装完整 prompt"""

    # 取当前状态对应的行为规则
    global_rules = case["customer_behavior_rules"].get("global", [])
    state_rules = case["customer_behavior_rules"].get("by_state", {}).get(current_state, [])
    all_rules = global_rules + state_rules

    parts = [
        Path("core/prompts/base_system.md").read_text(encoding="utf-8"),
        f"## 你扮演的客户\n{json.dumps(case['customer_role_card'], ensure_ascii=False, indent=2)}",
        f"## 当前状态: {current_state}\n规则:\n" + "\n".join(f"  - {r}" for r in all_rules),
        build_few_shot_string(case),
        f"## 隐藏信息\n{json.dumps(case['hidden_customer_state'], ensure_ascii=False, indent=2)}",
        f"## 最近对话\n{format_history(history[-6:])}",
        "## 输出格式\n返回 JSON: {customer_reply, next_state, triggered_events, score_notes}"
    ]
    return "\n\n".join(parts)
```

## 七、核心结论

1. **`roleplay_cases.jsonl`**: 最需要改进——加 `state_machine`（让客户行为有状态连续性）和 `few_shot_examples`（让模型真正见过这个客户怎么说话）。这是让陪练系统变聪明的**最快路径**。

2. **`evaluation_rubrics.jsonl`**: 评分标准太定性——加 `observable_evidence`（可观测证据锚点）和 `stage_specific_weights`（不同阶段不同权重）。

3. **模型关联**: 三个文件现在是静态文档，缺 `prompt_assembler.py` 把它们在每次 LLM 调用时动态注入。这是从「有数据」到「数据驱动模型」的**最后一公里**。
